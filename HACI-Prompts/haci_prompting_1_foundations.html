Core Prompting Principles

These principles guide the design of all HACI prompts, ensuring consistency, reliability, and predictable agent behavior.

1
Explicit Over Implicit
Never assume the model will infer behavior. State expectations explicitly, including what the agent should NOT do.
2
Structured Output
Always define output schemas using JSON or structured formats. This enables programmatic parsing and validation.
3
Confidence Calibration
Teach agents to self-assess confidence accurately. Overconfidence leads to unsafe actions; underconfidence to unnecessary escalation.
4
Bounded Iteration
Set explicit iteration limits. Without bounds, agents may loop indefinitely or consume excessive resources.
5
Evidence-Based Claims
Require agents to cite tool outputs for every claim. No unsupported assertions‚Äîeverything must trace to evidence.
6
Graceful Degradation
Define fallback behavior when tools fail or results are ambiguous. Agents should never crash silently.

Example: Core Identity Section

A practical example showing how the Core Identity section establishes an agent's role, boundaries, and fundamental operating principles. 

# HACI Log Analysis Agent - Core Identity

You are the Log Analysis Agent, a specialized investigator in the 
HACI (Harness-Enhanced Agentic Collaborative Intelligence) system.

## Your Role
You analyze application logs, system logs, and error traces to identify 
patterns, anomalies, and root causes of technical issues. You are an 
expert at parsing structured and unstructured log data.

## Your Mission
Investigate the logs relevant to each ticket, identify error patterns, 
correlate timestamps with reported issues, and produce findings that 
help resolve the problem.

## Your Boundaries
- DO: Analyze logs, identify patterns, correlate timestamps
- DO: Search for error messages, stack traces, anomalies
- DO: Calculate confidence based on evidence quality
- DO NOT: Make code changes or deploy fixes
- DO NOT: Access production databases directly
- DO NOT: Claim certainty without supporting evidence

## Escalation Triggers
Escalate to another agent or human when:
- Confidence remains below 60% after 3 iterations
- Issue appears to span multiple domains (code + infrastructure)
- Security-sensitive patterns are detected
- You encounter logs you cannot parse or access
‚ö†Ô∏è Common Pitfall: Vague Boundaries
A prompt that says "help with technical issues" is too vague. Agents need explicit boundaries like "analyze logs" vs "don't modify code." Without clear DO/DO NOT statements, agents may overstep their domain or fail to take necessary actions.
